# 帝国 CMS Codex 执行计划

> 本文档基于原始重构规划整理成 Codex 持续执行版本。
> 目标不是替代原方案，而是把它转换成“可连续开发、可追踪状态、可按依赖推进”的执行清单。

## 1. 执行原则

- 先完成功能骨架，再回头统一做视觉和交互细节。
- 优先推进能形成闭环的模块，避免只出现空菜单或半成品页面。
- 每一阶段都按“数据结构 -> 后台资源 -> 基础联动 -> 演示数据 -> 验证”的顺序推进。
- 不提前优化样式、品牌、复杂权限、复杂上传体验，除非它阻塞主线功能。
- 后续所有开发默认以本文档为主执行基线。

## 5. Codex 工作模式

后续默认按以下模式工作：

1. 先对照本文档确认当前所在阶段。
2. 只做当前阶段最关键、最依赖前置的数据和功能。
3. 每轮开发优先交付完整功能闭环，不优先做表面优化。
4. 每轮完成后更新“已完成 / 进行中 / 下一步”状态。

## 7. 2026-04 新阶段补充

### 当前共识

- 后台视觉体系已基本成型，前台仍处于“能显示内容，但没有成熟风格”的阶段。
- 前台必须先建立统一视觉系统，再继续首页、分类页、标签页、商店、订阅、活动等模块。
- 商店方向不继续锁死为“自造数字商品系统”。
- 当前新增的 `products / orders / payments` 只作为通用业务底座保留。
- 后续商店能力优先评估 Filament 官方生态或成熟插件，再决定是否接入实体商品流程。

1. 前台视觉系统与首页
2. 分类页与标签页
3. 站点配置中心补齐
4. 会员订阅、支付、订单底座
5. 商店方案评估与接入路径确认
