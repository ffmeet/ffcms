# 1.0 审计与安全复核

更新时间：2026-05-02

## 结论

产品主流程已经接近可发布，但以“生产观察版”标准来看，仍有几项需要明确处理：

- 必须修复或禁用的高风险项
- 可以带着上线、但要列入观察清单的中风险项
- 仅影响体验和运维效率的低风险项

## 高风险

### 1. 生产环境下存在模拟支付入口

位置：

- [routes/web.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/routes/web.php:52)
- [MemberOrderController.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/app/Http/Controllers/Web/MemberOrderController.php:59)

说明：

会员订单页原本允许登录用户对自己的订单直接提交 `paid / failed / closed` 模拟动作，并触发完整支付生命周期回写。  
这会直接影响订单、订阅、活动报名和库存状态。

当前处理：

- 已在控制器中加入 `production` 环境禁用。

建议：

- 正式支付接通前，前台不要保留任何用户可见的“模拟支付”操作。

### 2. webhook 缺少真实来源校验

位置：

- [routes/web.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/routes/web.php:82)
- [PaymentWebhookController.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/app/Http/Controllers/Web/PaymentWebhookController.php:15)

说明：

当前 webhook 逻辑只检查 `provider` 是否存在，然后记录 headers 与 payload，再把信息写入支付记录 payload。  
它还没有执行真实网关签名校验、时间戳校验或来源 IP 校验。

当前风险：

- 任何人都可以向公开 webhook 地址发送伪造诊断请求。
- 虽然当前代码还没有直接把订单标成已支付，但会污染支付审计数据和回调历史。

建议：

1. 正式接通网关前，将其明确标记为“诊断入口”，不要当正式支付回写使用。
2. 正式发布支付功能前，必须按支付渠道分别完成签名校验。

## 中风险

### 3. 登录、注册、找回密码接口原本没有限流

位置：

- [routes/web.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/routes/web.php:30)
- [AuthController.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/app/Http/Controllers/Web/AuthController.php:23)

说明：

这一组认证入口原本没有显式限流，容易被暴力尝试、撞库或邮件轰炸。

当前处理：

- 已给登录、注册、找回密码、重置密码提交路由补上基础限流。

建议：

- 后续再细化为按 IP 与账号维度的独立 RateLimiter。

### 4. 密码复杂度偏低

位置：

- [AuthController.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/app/Http/Controllers/Web/AuthController.php:91)

说明：

当前密码只要求 `min:6`，对生产环境偏弱。

建议：

- 提高到至少 8 位，并加入字母数字组合要求。

## 低风险 / 发布观察项

### 5. 首页主题位与 fallback 逻辑仍较复杂

位置：

- [HomeController.php](/Volumes/小芳侠/网站代码库/帝国 cms/ffmeet/laravel-ecms/app/Http/Controllers/Web/HomeController.php:18)

说明：

为了兼容旧数据和空配置，首页仍保留 fallback。  
这不直接构成安全问题，但会提高维护成本，也更容易出现“看起来没按后台配置出内容”的错觉。

建议：

- 生产稳定后，逐步把 `xiaofang` 首页改成“只吃明确位置位”的硬结构。

## 建议上线门槛

### 可以发布 1.0 观察版的条件

1. `APP_DEBUG=false`
2. 生产环境禁用模拟支付
3. SMTP 已配置可真实发信
4. webhook 不作为正式支付成功依据
5. 首页、会员中心、活动、商品、评论回归通过

### 不建议直接宣称“正式支付上线”的条件

1. 没有签名校验
2. 没有真实渠道联调
3. 没有支付异常补偿流程
