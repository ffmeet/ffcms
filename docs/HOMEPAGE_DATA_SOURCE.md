# 首页位置系统说明

> 适用主题：`xiaofang / 小芳侠`
>
> 目标：把首页调用逻辑从“模块文案命名”切换成“固定位置编号”，避免后续因为文案改名而影响数据映射与运营理解。

## 1. 为什么改成位置编号

首页里像“灵感”“继续阅读”“推荐展示”这类文案，本质上只是当前视觉方案里的标题或表达方式。

它们有几个问题：

- 文案会变
- 同一个位置未来可能换标题
- 不同主题的表达方式不一样
- 如果直接用文案给后台字段命名，后期很容易出现“名字变了，但位置没变”的混乱

所以现在首页设置改为：

- 使用固定位置编号 `01 / 02 / 03 / 04 / 05`
- 每个编号只代表一个**结构位置**
- 文案标题仍然可以自由调整
- 数据来源和结构位置解耦

## 2. 当前实现结构

后台已新增独立的“首页设置”页：

- `app/Filament/Pages/HomepageCenter.php`
- `resources/views/filament/pages/homepage-center.blade.php`

设置页不是按“灵感模块”“继续阅读模块”来配，而是按：

- `01 号位`
- `02 号位`
- `03 号位`
- `04 号位`
- `05 号位`

同时页面内已经加入虚拟位置图，帮助运营和开发对应首页结构。

## 3. xiaofang 首页位置定义

当前 `xiaofang` 首页已抽象为 5 个位置位：

- `slot_01`
  - 首屏左主位
- `slot_02`
  - 首屏右侧信息带
- `slot_03`
  - 第一屏右栏：最新文章列表
- `slot_04`
  - 第二屏左栏：文化卡片主区
- `slot_05`
  - 第二屏右栏：订阅入口与活动列表
- `slot_06`
  - 第三屏：创作灵感区
- `slot_07`
  - 第四屏：继续阅读区

这几个位置是结构定义，不依赖最终页面上显示成什么标题。

例如：

- `slot_04` 今天可以叫“创作灵感”
- 明天也可以改成“人物观察”
- 甚至后面改成“专题入口”

但它仍然是 `04 号位`。

## 4. 数据存储方式

首页位置配置是**和主题绑定**的，不同主题互不覆盖。

当前存储路径：

- `business_settings.theme_homepage.default.*`
- `business_settings.theme_homepage.editorial.*`
- `business_settings.theme_homepage.xiaofang.*`

`xiaofang` 主题当前使用：

- `business_settings.theme_homepage.xiaofang.slot_01.category_ids`
- `business_settings.theme_homepage.xiaofang.slot_02.category_ids`
- `business_settings.theme_homepage.xiaofang.slot_03.category_ids`
- `business_settings.theme_homepage.xiaofang.slot_04.category_ids`
- `business_settings.theme_homepage.xiaofang.slot_05.category_ids`

也就是说，每个位置保存的是：

- 可配置内容位：分类 ID 数组 + 取数上限 `limit` + 排序方式 `sort`
- 系统位（例如第二屏右栏）：当前只保留结构定义，不走文章分类设置

而不是一个固定文案名，也不是一个单独分类字段。

## 5. 为什么每个位置支持多个分类

这是这次优化的关键点。

之前如果一个位置只能绑定一个分类，会有几个限制：

- 一个区块内容容易单一
- 某个分类内容不够时，模块会很空
- 一个栏目很难同时承担多个前台结构位

现在每个位置支持：

- 选择多个分类
- 单独控制该位置取几条
- 单独控制该位置的排序优先级

这意味着：

- `slot_02` 可以同时取“人物 + 展览 + 阅读”
- `slot_04` 可以同时取“文化 + 观察”
- `slot_05` 可以同时取“生活方式 + 随笔 + 专题”

同时，同一个分类也允许被多个位置复用。

例如：

- “文化”可以同时参与 `slot_02`
- 也可以同时参与 `slot_04`

这就解决了你提到的：

> 一个分类下需要调用多个模块

这个场景。

## 6. 前台首页当前如何取数

前台首页入口：

- `app/Http/Controllers/Web/HomeController.php`

当前 `xiaofang` 首页已经优先改为按位置位读取分类：

- `slot_01`
- `slot_02`
- `slot_03`
- `slot_04`
- `slot_05`

控制器通过：

- `SiteTheme::homepageSlotCategoryIds(...)`

读取对应位置的分类数组，再按分类查询文章。

当前规则：

1. 如果该位置已配置分类
   - 优先按这些真实分类取数
   - 并按该位置自己的 `limit / sort` 输出结果
2. 如果该位置尚未配置
   - 自动回退到旧的首页精选池逻辑

这样做的好处是：

- 不会因为后台还没配完就导致首页空白
- 可以平滑从测试数据切到生产数据

## 7. 后台操作规则

现在后台“首页设置”的正确理解方式是：

- 不要把它理解成“设置某个文案模块”
- 要理解成“为某个位置编号绑定分类来源”

运营配置时建议：

1. 先看位置图
2. 确认要配置的是 `01 / 02 / 03 / 04 / 05` 哪个区块
3. 再为这个位置绑定 1 个或多个分类

这样即使后续首页文案、标题、视觉结构略有变化，后台配置仍然稳定。

## 8. 缓存机制

当前首页相关缓存已经有两层：

### 8.1 设置缓存

- `SiteSetting::current()` 走缓存
- 保存设置后自动失效

相关文件：

- `app/Models/SiteSetting.php`

### 8.2 首页结果缓存

首页控制器结果已经做了短时缓存，缓存键会带上：

- 当前主题
- 首页数量设置
- 当前主题首页位置配置

相关文件：

- `app/Http/Controllers/Web/HomeController.php`
- `app/Support/FrontendCache.php`

另外现在保存“首页设置”时，会主动刷新一次前台缓存版本：

- 不只清理站点设置缓存
- 也会立即让首页结果缓存失效
- 避免后台刚改完位置位，但前台还短时间停留在上一版结果

另外后台已新增简洁的“一键清理缓存”入口：

- `app/Filament/Pages/CacheCenter.php`
- `resources/views/filament/pages/cache-center.blade.php`

## 9. 结论

当前首页设置已经升级为：

- 按主题隔离
- 按位置编号管理
- 支持虚拟位置图
- 每个位置支持多个分类
- 同一分类可服务多个位置
- 前台已优先按真实分类取数
- 未配置时仍可回退旧逻辑

因此后续再调整首页文案时：

- 不需要改动位置定义
- 不需要因为标题变更去改字段名
- 只需要在后台继续维护位置编号和分类来源即可
